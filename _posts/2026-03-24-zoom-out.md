Sometimes, we need to change the way we see things.

Code is no different.

Throughout my experience working in teams, I’ve noticed a recurring pattern: there’s always some kind of implicit rule, some “secret standard” that no one has written down, but everyone somehow follows.

And most of the time, it works.

Stable teams develop their own shortcuts. Decisions live in people’s heads. Context is naturally shared… until it isn’t.

You start to notice it in small details.

Like how we handle migrations: we never merge a migration file with conflicts. If that happens, we accept the version from `main` and generate a new one.

Or in UI decisions: if you need a layout with flex or grid, we tend to prefer Tailwind over MUI. Not because it’s mandatory, but because, in our context, it usually leads to cleaner code.

None of these are wrong.

They’re actually good decisions and ideally they should be documented. But expecting 100% of these patterns and conventions to be written down isn’t realistic. At least, not right now.

The problem is that they remain invisible.

And over time, invisible things tend to accumulate.

I’ve seen this happen in a project that has been evolving since 2015. It became a kind of snowball: 147 unique contributors, multiple teams, different contexts, and shifting priorities.

As an internal tool, it grew organically. Ownership changed hands many times, and with that, understanding became distributed across people instead of centralized in documentation.

Some of that knowledge is still here.  
Some of it left with people who moved on.

The patterns and conventions exist, but they’re fragmented.

Similar problems were solved in different ways. Different teams made different trade-offs. And while none of that is inherently wrong, it introduces friction.

Onboarding slows down.  
Decisions take longer.  
Inconsistencies start to appear.

And when you try to build something meant to be shared across teams…

You start to feel the weight of all those invisible decisions.

This is where things begin to shift.

The way we build software is changing. Tools like Claude, and the idea of “skills”, are becoming part of the development workflow.

That creates an opportunity.

What if those implicit rules didn’t have to stay implicit?

What if patterns and conventions could be applied consistently while we code and reinforced during code reviews?

Instead of relying only on memory, or hoping that “someone will catch it,” we can start encoding these decisions.

Making them visible.  
Repeatable.  
Shareable.

Not perfectly. Not all at once. But better than before.

Because in the end, the problem isn’t that conventions live in people’s heads.

It’s depending on that.

Sometimes, improving a system doesn’t start with adding more rules.

It starts with changing perspective.

Zoom out.

If it only works because people **know**, it won’t work when people **change**.
